schillix at mailcan.com wrote:

> What caught my attention about Schillix is that there's a place to
> choose between 32-bit and 64-bit, and I'd like to know some more.
>
> Does this mean that I can have a purely 64-bit system, including libs &
> everything? This is of interest since I recall seeing a whole lot of
> i386 in Solaris 10 u8.

Why do you like to have a "pure" 64 bit systgem?

> From what I understand, which is quite possibly misunderstood, that
> i386, i486, i586 & i686 all refer to commands apps send to the kernel,
> with i386 being the simplest for old mobos from back when 1M RAM was a
> big deal, & i686 being able to talk to dual-socket Pentium 3's.
>
> I vaguely understand that the _64 suffix to x86 refers to something
> called an extension. I imagine it's like an adapter to be able to count
> higher. And that this ability to count higher allows for more to be done
> in 1 CPU cycle.
>
> Wouldn't it be a waste of hardware to have x86 software run on an x86_64
> CPU? 

It is called binary compatibility and helps to avoid a lot of other problems
if you like to support both.

> Is it an erroneous assumption that a system comprised of entirely x86_64
> software run more efficiently than a mixed bag?

On a typical 64 bit architecture, typical 64 bit binaries run 10-20% slower
than 32 bit applications. the AMD64 architecture is the only exception and this
results from doubling the amount of rregisters in 64 bit mode. A portable OS
does not depend on exceptions like the one for AMD64.
